# Project Charter

## Project Title
Secure Infrastructure Design, Deployment, Defense and Assessment for a Remote-first Startup

## Project Type
Final CYBFS Infrastructure and Cybersecurity Project

## Project Overview
This project consists of designing, documenting, deploying, securing, validating, and presenting a realistic SME-scale infrastructure for a remote-first startup.

The infrastructure is intended to support remote users through centralized identity and access management, a web service, a protected database, a file-sharing platform, monitoring, backup, and baseline security controls.

The project also includes a defensive and offensive review dimension through Blue Team and Red Team deliverables aligned with the deployed environment.

## Project Context
The selected company profile is a remote-first startup. This type of organization depends on secure remote access, centralized administration, collaborative digital services, and controlled access to internal resources.

The company needs a realistic infrastructure that remains:
- functional
- secure
- integrated
- manageable
- documented
- defensible during final review and presentation

## Project Problem Statement
Without a structured infrastructure, the company faces several risks:
- uncontrolled remote access
- fragmented account and permission management
- weak visibility over services and incidents
- poor control over shared files and application data
- weak recovery capability in case of failure or data loss
- excessive exposure of critical technical services

The project aims to solve these issues by proposing and documenting a coherent, secure, and demonstrable infrastructure adapted to SME constraints.

## Project Objectives

### Main Objective
Design and deliver a realistic, secure, and documented infrastructure for a remote-first startup.

### Detailed Objectives
- deploy centralized identity and access management
- deploy a web service layer
- deploy a protected backend database
- deploy a collaborative file-sharing service
- implement backup and monitoring logic
- implement baseline security controls
- document the infrastructure and its design choices
- validate service functionality and integration
- produce Blue Team and Red Team deliverables
- prepare a final professional presentation

## Project Scope

### In Scope
The project includes:
- business context and PRD documentation
- technical specification of the architecture
- infrastructure deployment documentation
- AD / IAM component
- Web Server component
- Database component
- File Server component
- Backup and Monitoring component
- Network Security component
- deployment evidence and screenshots
- project management artifacts
- Blue Team documentation
- Red Team documentation
- technology watch report
- final presentation

### Out of Scope
The project does not aim to include:
- full enterprise-scale infrastructure
- advanced multi-site failover
- full SOC maturity
- advanced zero-trust enforcement at scale
- enterprise-grade business continuity guarantees
- complete regulatory compliance implementation
- large-scale infrastructure automation and orchestration

## Expected Deliverables
The project is expected to produce the following deliverables:
- global project README
- context and PRD documentation
- technical specification document
- infrastructure deployment documentation
- project management file
- Blue Team deliverables
- Red Team deliverables
- technology watch report
- final presentation
- evidence and appendices

## Technical Scope
The infrastructure must include the following core technical elements:
- centralized identity and access management
- one web server or web application
- one protected database
- one file-sharing or collaboration service
- backup logic
- monitoring visibility
- firewalling and access control
- controlled remote access
- basic defensive readiness

## Success Criteria
The project will be considered successful if:
- all core services are deployed or clearly demonstrated
- service integration is functional
- access control is centralized and coherent
- service exposure is limited and justified
- backup and monitoring are documented and evidenced
- Blue Team and Red Team outputs are coherent with the deployed environment
- documentation is professional and usable
- the final presentation clearly explains the architecture, decisions, risks, and lessons learned

## Constraints

### Time Constraints
- limited project timeframe
- need to prioritize a realistic MVP
- need to balance build, documentation, testing, and presentation

### Resource Constraints
- SME-scale assumptions
- limited infrastructure resources
- project-oriented deployment environment
- limited complexity compared with enterprise production systems

### Security Constraints
- critical services must not be unnecessarily exposed
- backend services must remain protected
- remote access must be controlled
- administrative access must be restricted and documented

### Documentation Constraints
- all major choices must be explainable
- the infrastructure must remain demonstrable
- evidence must be collected and organized

## Project Assumptions
The project assumes:
- a small-to-medium remote-first startup scenario
- a manageable number of users and services
- a hybrid or virtualized lab-style deployment model
- realistic but simplified operations compared with a production enterprise environment
- enough time to complete a coherent MVP and its documentation

## Risks to Project Delivery
The main risks affecting project delivery include:
- time overrun due to excessive scope
- incomplete deployment evidence
- partial integration between services
- insufficient backup or monitoring validation
- weak linkage between technical deployment and Blue/Red Team outputs
- underdeveloped project management artifacts
- incomplete preparation for final presentation

## Risk Mitigation Approach
To reduce delivery risk, the project follows these principles:
- start with a clearly defined MVP
- prioritize the minimum required technical components
- document as deployment progresses
- validate core services early
- keep architecture understandable
- preserve evidence continuously
- align Blue Team and Red Team outputs with the real environment
- prepare final presentation from documented material rather than from memory

## Stakeholders

The project follows a simple delivery logic:

The project can be structured around the following milestones:

## Project Value
The project demonstrates the ability to:
- understand business needs
- translate them into technical architecture
- deploy integrated services
- apply cybersecurity thinking to infrastructure
- document and validate a realistic environment
- communicate technical work professionally

## Conclusion
This project charter defines the purpose, scope, objectives, constraints, and expected value of the final CYBFS project.

It positions the work as a realistic SME-scale infrastructure and cybersecurity project for a remote-first startup, with a strong focus on coherence, documentation, security, and demonstrability.
