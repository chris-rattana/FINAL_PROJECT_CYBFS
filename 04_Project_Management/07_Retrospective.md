# Retrospective

## Purpose
This document provides a retrospective view of the final CYBFS project.

Its purpose is to:
- reflect on how the project was conducted
- identify what worked well
- identify what was difficult or incomplete
- capture lessons learned
- show continuous improvement logic in the project management approach

This retrospective supports the Agile and project management dimension of the final project by demonstrating that the work was not only executed, but also reviewed critically and professionally. :contentReference[oaicite:0]{index=0}

## Retrospective Scope
This retrospective covers the main workstreams of the project:
- project framing and scoping
- technical specification
- infrastructure deployment
- security implementation
- testing and validation
- Blue Team and Red Team alignment
- documentation
- presentation preparation

## What Went Well

### 1. Project Scope Became Clearer Over Time
One of the main strengths of the project was the progressive clarification of scope.

At the beginning, the project could easily have become too broad because the final subject covers:
- infrastructure design
- deployment
- security
- documentation
- project management
- pentest logic
- final presentation

The use of an MVP-oriented approach helped reduce this risk and keep the project centered on the required core components. This was aligned with the project guidance encouraging teams to start small, reduce scope, and focus on something working rather than something oversized. :contentReference[oaicite:1]{index=1}

### 2. The Documentation Structure Became a Strong Point
A major positive point of the project was the progressive creation of a coherent repository structure.

This helped:
- organize deliverables clearly
- separate technical design from deployment proof
- distinguish Blue Team and Red Team perspectives
- prepare the final project in a more professional way
- reduce the risk of losing evidence

The repository structure became one of the project’s strengths because it made the work more traceable and easier to review.

### 3. The Infrastructure Was Approached as an Integrated System
Another strong point was treating the infrastructure as a coherent whole rather than as isolated technical components.

This improved the logic of:
- identity and access management
- web and database integration
- file-sharing access control
- backup and monitoring coverage
- network security
- Blue Team and Red Team linkage

This helped keep the final project closer to the actual expectations of the subject, which emphasizes a fully integrated and secure infrastructure rather than disconnected technical demonstrations. :contentReference[oaicite:2]{index=2}

### 4. Security Was Integrated Early
Security was not treated only at the end of the project.

It was progressively included through:
- access control logic
- firewalling and exposure limitation
- protected backend services
- backup and monitoring thinking
- Blue Team and Red Team preparation
- risk analysis and security controls documentation

This improved the consistency between deployment and security review.

### 5. The Project Management Layer Added Value
The project management folder added real value by making the project more complete and professional.

Instead of treating the work only as a technical lab, the project now includes:
- charter logic
- roles and responsibilities
- Agile practices
- Kanban evidence
- roadmap and budget placeholders
- retrospective thinking

This makes the final project more aligned with the expected deliverables. :contentReference[oaicite:3]{index=3}

## What Was Difficult

### 1. Scope Control Was a Constant Challenge
The biggest difficulty of the project was the natural tendency to make the environment too ambitious.

The subject is broad and invites many possible additions:
- advanced hardening
- advanced monitoring
- richer backup models
- more complex networking
- deeper pentest scope
- more polished management artifacts

Without active scope control, the project could easily become too large to finish cleanly.

### 2. Turning Templates into Final Deliverables Took More Work Than Expected
A major challenge was that having a good framework was not the same as having a finished deliverable.

At several points, the project had:
- strong structure
- strong templates
- strong methodology

but not yet enough:
- real completed evidence
- real completed case material
- fully populated final documents

This showed the difference between “being prepared to deliver” and “having actually delivered.”

### 3. Evidence Collection Needed More Discipline
Another difficulty was evidence collection.

It is easy during technical work to focus on configuration and forget to preserve:
- screenshots
- service status outputs
- access-denied proof
- integration proof
- backup proof
- firewall proof

This created a risk of having working services without enough material to support final validation and presentation.

### 4. Documentation Volume Became Significant
The project required a large amount of documentation:
- context and PRD
- technical specification
- deployment documentation
- screenshots and proofs indexes
- project management documentation
- Blue Team and Red Team deliverables
- technology watch report
- presentation preparation

This made documentation itself a significant workstream, not just a side task.

### 5. Final Coherence Across All Parts Required Care
Another challenge was maintaining coherence between:
- business context
- architecture
- deployment
- security controls
- risk analysis
- Blue Team findings
- Red Team findings
- project management
- final presentation

The more the project grows, the more important coherence becomes.

## What Could Be Improved

### 1. Collect Evidence Earlier and More Systematically
In a future iteration, evidence collection should happen even earlier and more systematically.

A better approach would be:
- take screenshots at the moment of validation
- save key command outputs immediately
- name evidence files consistently from the start
- reference evidence while writing deployment notes

This would reduce the risk of missing proof later.

### 2. Freeze the MVP Earlier
Another improvement would be to define and freeze the MVP even earlier in the project.

This would help:
- reduce unnecessary branching
- avoid spending time on optional enhancements too early
- improve planning accuracy
- focus attention on required deliverables first

### 3. Create the Kanban Board Earlier
Although Kanban is highly useful for tracking work visually, it is most effective when used from the very beginning.

Creating and updating it earlier would improve:
- task visibility
- ownership clarity
- progress tracking
- bottleneck identification

This is directly aligned with the project guidance on Kanban as a practical way to visualize and manage work. :contentReference[oaicite:4]{index=4}

### 4. Validate Integration Earlier Between Components
Another improvement would be to validate component integration earlier instead of waiting until several services are already built.

Earlier checks between:
- web and database
- IAM and file-sharing
- firewall rules and service exposure
- monitoring and critical services

would reduce late-stage surprises.

### 5. Prepare Presentation Material Progressively
A future improvement would be to extract presentation-ready visuals and summaries progressively during the build.

This would reduce the final pressure of:
- rebuilding the project story at the end
- searching for proof late
- compressing too much information into a short preparation window

## Key Lessons Learned

### Lesson 1 — Good Structure Is a Force Multiplier
A clear repository structure and strong document organization make a big difference in project quality.

### Lesson 2 — Working Infrastructure Is Not Enough Without Evidence
A service that works but cannot be proven, explained, or validated is weaker in a final project context.

### Lesson 3 — Security Must Be Built Into the Architecture
Security is easier to justify and demonstrate when it is integrated from the beginning rather than layered on at the end.

### Lesson 4 — Scope Discipline Is Essential
A realistic, complete, and coherent project is more valuable than an oversized but unfinished one.

### Lesson 5 — Project Management Improves Technical Delivery
Planning, task tracking, role definition, and retrospective thinking genuinely improve technical outcomes.

## What Should Be Kept in Future Projects
The following practices should clearly be kept:
- MVP-first thinking
- structured repository organization
- documentation written progressively
- separation between component description, installation notes, and evidence
- early security reasoning
- Blue Team and Red Team alignment with the real environment
- project management artifacts integrated into the technical work

## What Should Change in Future Projects
The following practices should be improved in future projects:
- earlier Kanban use
- stricter evidence collection discipline
- earlier integration testing
- faster distinction between required deliverables and optional improvements
- earlier presentation preparation

## Overall Assessment
Overall, the project progressed in a positive and professional direction.

Its main strengths are:
- strong structure
- coherent documentation logic
- realistic infrastructure thinking
- integrated security perspective
- improved project management maturity

Its main difficulties were:
- scope management
- converting templates into fully completed deliverables
- maintaining consistent evidence collection
- keeping all project parts aligned as the repository expanded

## Conclusion
This retrospective shows that the project was not only executed, but also critically reviewed.

The main outcome is that a realistic and professionally structured final project depends on more than technical build quality alone. It also depends on scope control, documentation discipline, evidence management, and continuous alignment between infrastructure, security, project management, and communication.

These lessons strengthen the credibility of the final project and provide a useful basis for improvement in future infrastructure and cybersecurity projects.
